  <name>Task 1: Create syllabus parser prompt template</name>
  <files>src/lib/llm/prompts/syllabus-parser.ts</files>
  <action>
Create a prompt module following the goal-parser.ts pattern. The prompt should:

1. Define TypeScript interfaces for the parsed output:
   - `ParsedSyllabusAssignment`: title, type (homework/exam/project/reading/quiz/paper/other), dueDate (YYYY-MM-DD), estimatedHours, notes (optional)
   - `ParsedSyllabus`: courseName, instructor (optional), semester (optional), assignments array
   - `SyllabusParserResponse`: union of ParsedSyllabus or ClarificationNeeded

2. Build system prompt (`buildSyllabusParserPrompt`) that:
   - Instructs LLM to extract course info and ALL assignments from syllabus text
   - Returns JSON with course details + assignments array
   - Hour estimation heuristics embedded in prompt:
     * Homework: 1-3 hours (simple) to 5-8 hours (complex)
     * Reading: ~30 pages/hour, chapters typically 20-40 pages
     * Quiz: 0.5-2 hours prep
     * Exam: 4-10 hours prep (midterm ~6h, final ~10h)
     * Paper: 1 hour per page for writing + research time
     * Project: 10-30 hours depending on scope
   - Handle partial syllabi (missing dates = null, missing info = ask for clarification)
   - Return clarification questions if syllabus is too ambiguous

3. Create response parser (`parseSyllabusResponse`) that:
   - Extracts JSON from code block (like goal-parser.ts)
   - Validates each assignment has required fields
   - Validates dates are valid YYYY-MM-DD format
   - Returns null if parsing fails

4. Create helper `isSyllabusClarificationNeeded` type guard

Follow exact pattern from goal-parser.ts for structure, JSON extraction regex, and validation approach.

<task type="auto">
  <name>Task 3: Create syllabus parse API endpoint</name>
  <files>src/app/api/syllabus/parse/route.ts</files>
  <action>
Create POST `/api/syllabus/parse` following the parse-goal/route.ts pattern exactly:

1. Authentication: Get user from Supabase auth
2. Rate limiting: 5 requests per minute (syllabus parsing is heavier than goal parsing)
3. Token budget check: Use checkCombinedBudget like parse-goal
4. LLM status check: Fast fail if unavailable
5. Request validation: Parse body with syllabusParseRequestSchema
6. Build messages: System prompt from buildSyllabusParserPrompt + user message with syllabus text
7. Cache check: Generate cache key, check for cached response
8. LLM call: Use getLLMProvider().chat() with maxTokens: 2000 (syllabi can generate long responses), temperature: 0.2 (need accuracy)
9. Token tracking: Record usage to daily and session trackers
10. Cache response
11. Parse response: Use parseSyllabusResponse
12. Return parsed syllabus or clarification questions

Error handling:
- 401 for unauthenticated
- 429 for rate limit or budget exceeded
- 400 for invalid request body
- 503 for LLM unavailable
- 500 for parse failures (with helpful message to try shorter sections)

Response format:
```json
{
  "course": { "name": "...", "instructor": "...", "semester": "..." },
  "assignments": [{ "title": "...", "type": "...", "dueDate": "...", "estimatedHours": ... }],
  "provider": "anthropic",
  "cached": false
}
```

Or for clarification:
```json
{
  "needsClarification": true,
  "questions": ["..."],
  "provider": "anthropic"
}
```
