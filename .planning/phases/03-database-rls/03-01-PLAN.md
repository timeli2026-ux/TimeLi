<task type="auto">

  <files>docs/rls-policy-patterns.md</files>
  <action>
Create documentation for consistent RLS policy creation on future tables:

**Document these patterns:**

1. **User-owns-row pattern:**
   ```sql
   auth.uid() = user_id
   ```
   Use when: Table has direct user_id foreign key

2. **User-owns-via-profile pattern:**
   ```sql
   auth.uid() = (SELECT id FROM profiles WHERE id = table.profile_id)
   ```
   Use when: Table references profiles table

3. **Public-read-user-write pattern:**
   - SELECT: true (or authenticated)
   - INSERT/UPDATE/DELETE: auth.uid() = user_id
   Use when: Data is publicly viewable but user-editable

4. **Soft-delete pattern:**
   Include `deleted_at IS NULL` in SELECT policies
   Use when: Table uses soft deletes

**Checklist for new tables:**
- [ ] RLS enabled: `ALTER TABLE table_name ENABLE ROW LEVEL SECURITY;`
- [ ] SELECT policy created
- [ ] INSERT policy created (unless trigger-handled)
- [ ] UPDATE policy created
- [ ] DELETE policy created (or document why not needed)
- [ ] Policies use auth.uid() for user identification
- [ ] Policies tested with different user contexts

**Security notes:**
- Service role key bypasses RLS - never expose to clients
- SECURITY DEFINER functions bypass RLS - use sparingly
- Test policies with actual user sessions, not service role
